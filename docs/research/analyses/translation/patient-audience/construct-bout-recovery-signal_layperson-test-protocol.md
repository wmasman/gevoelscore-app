## 1. Purpose + status

### 1.1 What this protocol tests

The layperson-test verifies that a **layperson-reader** — a person outside the research line, with no methodology briefing and no researcher context — reads the patient-audience artefact as the drafter intended, on four surfaces plus the visual caption per guide #6 [`translation_to_audience.md`](../../../methodology/translation_to_audience.md) r3 §5.7.1 addendum:

- **Surface 1 — §5.3 core-claim** ("moment waarop mijn stress omhoog piekte en niet terugzakte" descriptive framing + tier-2 patroon-dat-je-achteraf-herkent framing + AGREES-on-direction-not-magnitude discipline).
- **Surface 2 — §5.4 refusals** (four "wat ik er NIET mee kan" categories: predictive-alert / causal / group-level / advice + extra tier-2 grenzen).
- **Surface 3 — §5.6 PPV frame** ("24 van de 25 keer fout" plain-Dutch parallel + substantive-interpretation-not-critique nuance + era-heterogeneity discipline).
- **Surface 4 — §5.11 subject-narrative commentary** (first-ever filled patient-audience commentary in the corpus; felt-experience anchor per Willem-approved "vastzitten" register; all four §3.12 hard separations + cascade-source bounded qualifier in plain-Dutch equivalent).
- **Plus the visual caption** — currently PENDING per §5.8; caption itself layperson-test-gated per guide #6 §5.8.

### 1.2 What passing means, per surface

A **PASS** on a surface means the layperson's paraphrase + probe response matches the drafter's intended reading of that surface. A **FAIL** means the layperson's reading imports one or more of the specific fail-modes enumerated in §4 below — forecast import, advice import, jargon import, causal-mechanism import, substitution-for-own-body-signal-interpretation import, patronising-read import.

**Definitive lock**: patient-audience artefact moves from LOCKED r2 with LAYPERSON-TEST-PENDING → LOCKED r3 with LAYPERSON-TEST-CLOSED when all four surfaces + visual caption pass. **Any surface FAIL** triggers §6.4-style revision per guide #6 §6.4 layperson-test-fail revision, followed by a fresh layperson-test round on the revised surface(s). Legitimate lock-state per guide #6 §5.7 is preserved during PENDING.

## 2. Layperson-tester recruitment scope

### 2.1 Recruitment binding

Per locked-plan [`_plan_results_analysis_layer.md`](../../../methodology/_plan_results_analysis_layer.md) §10.7 per-artefact pool resolution: **pool identification is the user's responsibility**, not delegated to Claude. This section names candidate tester profiles + minimum tester count + selection anti-patterns; specific tester identification and recruitment logistics are the user's tempo.

### 2.2 Valid tester profiles

Three candidate profiles are pre-declared per target artefact §5.2 audience-definition + guide #6 §4.4 audience-specificity binding:

- **Profile A — PAIS-peer with Garmin**: a person ≥2 years post-onset with a PAIS-family condition (Long COVID, ME/CFS, or similar), Dutch-speaking, comfortable reading lay-press health articles, currently owns or has owned a Garmin wearable device. This is the **primary target audience** per §5.2 first sentence.
- **Profile B — non-PAIS family member of a PAIS patient**: a person who has lived alongside a PAIS-condition household member, Dutch-speaking, no direct methodology background, curious about what the wearable data shows. This is the **secondary target audience** per §5.2 second sentence.
- **Profile C — non-specialist clinician (huisarts / fysiotherapeut) without deep ME/CFS methodology background**: a Dutch-speaking GP or PT who sees PAIS patients occasionally, no direct N-of-1 methodology background, curious about how to talk about wearable data in a clinic conversation. This is the **tertiary target audience** per §5.2 second sentence.

**Preference ordering for precedent-establishing first execution**: Profile A > Profile B > Profile C. Profile A is the primary target; testing on the primary audience is most informative. Profile B and C are ancillary validations.

### 2.3 Minimum tester count

- **Minimum**: 1 tester (Profile A preferred). This is the guide #6 §5.7 legitimate-lock-state minimum for closing LAYPERSON-TEST-PENDING.
- **Recommended for precedent-establishing first execution**: ≥2 testers (at minimum: 1 Profile A + 1 Profile B or C). Two-tester coverage strengthens the precedent baseline for future §5.11-carrying tracks and provides triangulation on ambiguous reads per §1.4.
- **Higher-tester option**: 3-4 testers (Profile A + Profile B + Profile C) for maximum precedent-strength. Not required. Left as user tempo.

### 2.4 Ambiguous-read handling

If a single tester's read is ambiguous per §1.4 (paraphrase partly matches drafter intent + partly diverges without a clear enumerated fail-mode signal), recruit a **second tester of the same profile** as the original tester and re-run the test on the ambiguous surface(s) only. Two independent testers of matched profile must both PASS the surface before it is treated as PASS. Do not average partial signals; do not "second-tester-override" a clean FAIL from tester 1.

### 2.5 Anti-patterns in recruitment

- **NOT valid**: any tester who has read the target artefact or its upstream sources (Stage A `construct-bout-recovery-signal.md`; Stage I `HA-C4c.md`; hypothesis + result `hypotheses/HA-C4c/*`). Prior exposure invalidates the cold-read baseline.
- **NOT valid**: any tester who has been briefed on the methodology framework (tier-1 / tier-2 / tier-3; PPV; N-of-1; L-IDs; verdict vocabulary). Methodology-briefing invalidates the layperson baseline.
- **NOT valid**: any tester whose Dutch is not native or fluent at daily-conversation level. Non-fluent Dutch invalidates the base-language target per §5.2 base language declaration + guide #6 §4.3 heuristic.
- **NOT valid**: any tester with a family-relation to the participant (Willem) whose response would be inhibited by the researcher-relationship. Personal-relationship shield may distort the honest first-read.

### 2.6 Consent + privacy

The tester is informed that the artefact describes N-of-1 research on Willem's personal Long COVID health data and that their reading + paraphrase + probe response will be recorded (pseudonymised) per §5 template. No personal data of the tester is required beyond the tester profile identifier (A/B/C) + a pseudonymised identifier + the test date. The tester has the right to skip any question or discontinue at any point without explanation.

## 3. Materials to present

### 3.1 Reading sequence

The tester reads the four §5.X sections in the following fixed order:

1. §5.3 core-claim paragraphs 1-4 + closing sentence.
2. §5.4 refusals (all four "wat ik er NIET mee kan" categories + Extra grenzen op tier-2 niveau + Bevinding gebonden aan periode en apparaat + De "alleen-in-rust" leesfout).
3. §5.6 PPV frame (all three sub-sections: Hoe vaak zou een dagelijkse waarschuwing kloppen als ze afgaat + Waarom is dit géén reden om de bevinding weg te doen + Basispercentage verandert per periode).
4. §5.11 subject-narrative commentary (opening bold sentence + three paragraphs + parenthetical toelichting).

Optional — §5.8 visual caption when the visual is produced (currently visual-pending per §5.8). Until the visual lands, the caption is not tested; §5.8 remains an artefact-level open input separate from this LP-test protocol.

### 3.2 Cold-read discipline

- **NO researcher context provided**: the tester is not told what the "correct" reading is; is not told what fail-modes to avoid; is not told the tier-2 vs tier-3 distinction; is not shown any upstream methodology.
- **NO expected-outcome anchoring**: the tester is not told "we're checking if you read this as a forecast" — that would prime avoidance. The probes in §4 are constructed to elicit the tester's read without importing the drafter's intent.
- **Minimal framing**: the researcher (or facilitator) tells the tester: *"Ik ga je een tekst laten lezen van een onderzoeker die iets over zijn eigen gezondheid heeft uitgezocht. Ik ga je een paar vragen stellen over wat je in de tekst leest. Er is geen goed of fout antwoord; ik ben benieuwd wat jij eruit haalt."* No further framing.
- **Reading pace**: at the tester's own pace. If the tester wants to re-read a section before answering, that is permitted.

### 3.3 Facilitator role

The researcher (or a research-line-independent facilitator) presents the material + asks the probes. The facilitator may clarify Dutch vocabulary the tester is unsure about (e.g. "wat is 'stringente' hier?") but MUST NOT paraphrase the drafter's intended reading or explain the finding. If the tester asks "what does the researcher mean by X?" the facilitator responds: *"Wat lees jij zelf in de zin? Dat is wat ik wil weten."*

## 4. Four-surface comprehension checkpoints

For each of the four surfaces + visual caption (when produced), the facilitator asks: **(a) an open-ended paraphrase question first** (elicits the tester's spontaneous reading), then **(b) targeted probe(s)** (checks specific fail-modes without importing drafter intent). Signals for PASS and FAIL are enumerated per surface.

### 4.1 Surface 1 — §5.3 core-claim

**Open-ended paraphrase question (facilitator wording)**:

> "Kun je in je eigen woorden vertellen wat je hier leest? Wat is volgens jou de kern van wat de onderzoeker zegt?"

**Targeted probes (facilitator wording, one at a time)**:

- Probe 1 — forecast-import test: *"Denk je dat de onderzoeker met deze bevinding morgen kan voorspellen of hij een klap krijgt?"*
- Probe 2 — direction vs magnitude test: *"De onderzoeker zegt dat de richting van zijn bevinding overeenkomt met wat er in de literatuur staat, maar dat de omvang op groepsniveau iets anders is. Wat betekent dat volgens jou?"*
- Probe 3 — rarity vs shift test: *"Snap je waarom een klein verschil hier iets bijzonders is, of leest het als een klein verschil zoals bij alle verschillen?"*

**Signal for PASS on Surface 1**:

- Paraphrase includes "achteraf herkend patroon" or "iets uit vroeger data" or "terugkijk-observatie" without the tester importing "voorspelt", "waarschuwt", or "kondigt aan".
- Probe 1 response: variation on "nee, dat zegt hij ook nadrukkelijk niet".
- Probe 2 response: variation on "dat de richting klopt met de literatuur maar hij kan de sterkte niet vergelijken omdat de meetinstrumenten anders zijn".
- Probe 3 response: variation on "omdat het al zeldzaam is telt het meer dan bij iets dat elke dag gebeurt", OR the tester recognises the tail-uiteinde framing without further prompting.

**Signal for FAIL on Surface 1**:

- Paraphrase includes forecast wording ("hij kan hiermee voorspellen dat...", "een waarschuwing dat...", "hij weet nu dat morgen...").
- Probe 1 response: variation on "ja, denk het wel" or "wel een beetje" without visible uncertainty.
- Probe 2 response: variation on "dat de literatuur bevestigt dat zijn bevinding sterk is" (importing group-level magnitude the artefact explicitly disclaims).
- Probe 3 response: variation on "geen idee waarom dat verschil zou maken" AND tester does not read tail-context from surrounding sentences.

### 4.2 Surface 2 — §5.4 refusals

**Open-ended paraphrase question**:

> "In deze sectie zegt de onderzoeker een aantal keer 'ik kan dit NIET gebruiken voor X'. Wat mag hij volgens jou wel met deze bevinding doen, en wat mag hij niet?"

**Targeted probes**:

- Probe 1 — advice-import test: *"Als de onderzoeker morgen twee van deze momenten in zijn data ziet, zou hij dan volgens deze tekst moeten rusten of niet zwaar sporten?"*
- Probe 2 — causal-import test: *"Zegt de onderzoeker dat zware inspanning deze momenten veroorzaakt?"*
- Probe 3 — group-level-import test: *"Zou deze bevinding ook betekenen dat andere mensen met PAIS dezelfde momenten hebben?"*
- Probe 4 — bare-percentage test: *"De onderzoeker noemt ergens 'ongeveer 2 tot 5 procent van de keren'. Is dat volgens jou hoog of laag, en waarom?"*

**Signal for PASS on Surface 2**:

- Paraphrase includes at least 2 of the 4 refusal categories (predictive / causal / group-level / advice) without importing them as things the artefact endorses.
- Probe 1 response: variation on "nee, dat staat er ook niet, dit is geen instructie".
- Probe 2 response: variation on "nee, hij zegt expliciet dat het geen oorzaak-en-gevolg is".
- Probe 3 response: variation on "nee, dat kan hij hier niet uit afleiden, dat zou een groepsstudie vereisen".
- Probe 4 response: variation on "die 2 tot 5 procent moet je zien in de context van hoe zeldzaam crashes zijn (~2 per jaar), anders zegt het niets".

**Signal for FAIL on Surface 2**:

- Paraphrase: tester says the artefact IS an instruction (e.g. "hij zegt dat hij moet rusten als het gebeurt") or IS a causal explanation ("dus zware inspanning veroorzaakt die momenten").
- Probe 1 response: variation on "ja, hij zou moeten rusten" (advice-import).
- Probe 2 response: variation on "ja, de zware inspanning veroorzaakt de momenten" (causal-import).
- Probe 3 response: variation on "ja, dit geldt vast ook voor anderen met PAIS" (group-level-import).
- Probe 4 response: variation on "2 tot 5 procent is heel hoog" or "heel laag" without base-rate context (bare-percentage failure — reads the percentage without its context).

### 4.3 Surface 3 — §5.6 PPV frame

**Open-ended paraphrase question**:

> "De onderzoeker zegt dat een waarschuwing die op deze bevinding gebaseerd is, '24 van de 25 keer dat ze afgaat fout zou zijn'. Wat betekent dat volgens jou voor wat de onderzoeker eigenlijk heeft gevonden?"

**Targeted probes**:

- Probe 1 — substantive-interpretation-not-critique test: *"Betekent '24 van de 25 keer fout' volgens jou dat de bevinding zelf niet klopt?"*
- Probe 2 — era-heterogeneity test: *"De onderzoeker zegt ook dat in eerdere jaren, 2023-2024, hij ~10-11 crashes per jaar had, en in 2025-2026 ongeveer 2. Waarom noemt hij dat verschil?"*
- Probe 3 — structural constraint test: *"Denk je dat de onderzoeker deze bevinding zou weggooien omdat een waarschuwing zo vaak fout zou zijn, of houdt hij hem juist?"*

**Signal for PASS on Surface 3**:

- Paraphrase includes recognition that "24 van de 25 keer fout" is about the DAILY WARNING, NOT the underlying pattern. Tester reads the two as distinct.
- Probe 1 response: variation on "nee, dat zegt hij ook nadrukkelijk niet, het patroon zelf blijft geldig".
- Probe 2 response: variation on "omdat het basispercentage waarop je de waarschuwing meet, verandert per periode; dus wat 'kloppen' betekent hangt af van wanneer je meet".
- Probe 3 response: variation on "hij houdt hem, omdat het patroon zelf iets zegt over vroeger, ook al kan hij er geen goede waarschuwing van maken".

**Signal for FAIL on Surface 3**:

- Paraphrase: tester says the artefact IS retracting the finding (e.g. "dus deze bevinding klopt niet echt") OR the artefact IS endorsing a warning system (e.g. "dus 4 procent is genoeg om ermee te werken").
- Probe 1 response: variation on "ja, dus de bevinding klopt niet" (substantive-critique-import).
- Probe 2 response: variation on "omdat het gewoon niet werkt bij weinig crashes" without recognising the base-rate structural point.
- Probe 3 response: variation on "hij zou hem weggooien" (substantive-critique-import).

### 4.4 Surface 4 — §5.11 subject-narrative commentary

**Open-ended paraphrase question**:

> "In deze laatste sectie beschrijft de onderzoeker een persoonlijk gevoel dat hij bij deze bevinding heeft. Wat zegt hij volgens jou daar precies, en hoe verhoudt zich dat tot de bevinding in §5.3?"

**Targeted probes**:

- Probe 1 — forecast-import test on commentary: *"Zegt de onderzoeker in deze commentaar dat hij op een zware dag tegen het einde van de dag kan voorspellen dat hij een klap krijgt?"*
- Probe 2 — advice-import test on commentary: *"Adviseert de onderzoeker jou als lezer wat je zou moeten doen als je een 'vastzitten'-gevoel hebt?"*
- Probe 3 — substitution-for-own-body test: *"Als jij zelf een 'vastzitten'-gevoel hebt in je lichaam op een dag, wat zou de onderzoeker volgens deze tekst dan zeggen?"*
- Probe 4 — commentary-vs-formal test: *"De onderzoeker zegt dat deze commentaar 'geen advies' en 'geen instructie' is en dat hij 'het patroon niet sterker kan maken dan het is'. Wat betekent 'niet sterker maken' hier volgens jou?"*
- Probe 5 — jargon-in-context test: *"Het woord 'vastzitten' staat tussen aanhalingstekens. Waarom denk je?"*

**Signal for PASS on Surface 4**:

- Paraphrase includes recognition that the commentary is the researcher's OWN felt-experience (subject-attributed) and that the formal §5.3 pattern is the "gelicenceerde inhoud", with the commentary as a "lichaam-anker".
- Probe 1 response: variation on "nee, hij zegt expliciet dat het geen voorspelling is en dat het niet betekent dat er een crash komt".
- Probe 2 response: variation on "nee, hij adviseert helemaal niets, hij beschrijft alleen wat hij zelf voelt".
- Probe 3 response: variation on "dat mijn ervaring niet hetzelfde is als de zijne; hij zegt expliciet dat elke lezer haar eigen envelop bezit".
- Probe 4 response: variation on "dat de commentaar geen extra bewijs is voor het patroon; het is alleen zijn persoonlijke voelen ernaast".
- Probe 5 response: variation on "omdat het zijn eigen woord is voor hoe hij het voelt, geen wetenschappelijke term".

**Signal for FAIL on Surface 4**:

- Paraphrase: tester says the commentary IS predictive ("hij weet nu wanneer hij een klap krijgt") OR IS advice ("hij zegt dat je moet oppassen als je zo voelt") OR IS a claim about other people ("dus dit geldt ook voor mij als ik dit voel").
- Probe 1 response: variation on "ja, hij kan het voorspellen" (forecast-import on commentary — worst-case fail-mode).
- Probe 2 response: variation on "ja, dat lees ik ook zo" (advice-import on commentary).
- Probe 3 response: variation on "dus als ik dit voel dan heb ik hetzelfde als hij" (substitution-for-own-body-import).
- Probe 4 response: variation on "dat de commentaar het patroon versterkt" (commentary-as-evidence-import — violates §3.12 anti-substitution).
- Probe 5 response: variation on "geen idee" AND tester does not read the felt-language marker from surrounding sentences.

## 5. Response recording template

For each layperson-test session, the facilitator (or the researcher acting as facilitator) fills the following template. One template per tester per session.

### 5.1 Session metadata

- **Tester identifier**: `T{profile}{index}` — e.g., `TA1` for first Profile A tester, `TB1` for first Profile B tester. Pseudonymised. Do NOT record the tester's real name in this file. Tester's real name is kept in a separate off-repo record if user wants a mapping.
- **Tester profile** (A / B / C per §2.2).
- **Test date**: YYYY-MM-DD.
- **Facilitator identifier**: `Willem` (self-facilitated) OR `<pseudonymised third-party facilitator identifier>`.
- **Session duration**: HH:MM (approximate; useful for future protocol tuning).
- **Session mode**: `in-person` / `video-call` / `phone-call` / `written-async`. Written-async excludes probes that require live back-and-forth; note if used.

### 5.2 Per-surface recording

For each of the four surfaces (§5.3 + §5.4 + §5.6 + §5.11) plus the visual caption when produced, record:

- **Paraphrase (verbatim)**: the tester's spontaneous paraphrase in their own words, quoted verbatim. If in Dutch, keep Dutch; if translation is needed for analysis, add a bracketed EN gloss but keep Dutch as primary.
- **Probe responses (verbatim, per probe)**: the tester's response to each probe in order, quoted verbatim.
- **Facilitator assessment**: PASS / FAIL / AMBIGUOUS per §4 signals. If AMBIGUOUS, describe which specific signal component was unclear.
- **Notes**: any tester side-comment, non-verbal cue, or contextual information the facilitator observed that affects the reading.

### 5.3 Session summary

- **Overall session outcome**: `ALL-PASS` / `SURFACE-FAIL(s)` / `AMBIGUOUS` / `INCOMPLETE`.
- **Fail-mode summary** (if any surface FAIL): specific fail-mode + which probe(s) revealed it.
- **Ambiguity-triggered second-tester recommendation** (if any surface AMBIGUOUS): recommend recruit a second same-profile tester per §2.4.
- **Follow-up needed**: any content revision the facilitator recommends per guide #6 §6.4 layperson-test-fail revision, OR any protocol revision (this document) needed based on session experience.

Filled recording sessions land as:


One file per tester per session. Pseudonymised. Committed to the repo per audit_names.txt discipline (tester real names never enter recorded files).

- Recruit a second same-profile tester per §2.4.
- Re-run the test on the ambiguous surface(s) only.
- Two independent testers of matched profile must both PASS the surface before it is treated as PASS.
- If second tester's read is also AMBIGUOUS, treat as FAIL and route to §6.2 revision.
- If second tester's read is FAIL, treat first-tester read as a "close call" and route to §6.2 revision (asymmetric bar: ambiguity + fail from 2 testers = revision needed).
- If second tester's read is clean PASS, first-tester ambiguity is documented in aggregate outcome but the surface is treated as PASS (asymmetric bar preserved: single-tester ambiguity NOT enough to trigger revision when second tester is clean).
