# App audit — 2026-05-30

Four parallel domain audits run against the deployed code at `gevoelscore-frontend.fly.dev` (commit `93fb2a0` at audit time). Read-only — no code was changed during this audit.

Each audit was performed by an independent agent applying an external standard directly to the source tree, with the explicit instruction not to manufacture issues for thoroughness's sake. Findings cite `file:line` evidence; a sample (one finding per dimension) was spot-verified against the source after the audit completed.

- **Accessibility** — W3C WCAG 2.2 AA + Apple HIG (44×44 minimum) + the project's Brainfog cognitive-accessibility extensions
- **Security** — OWASP ASVS v4 L1+L2 + OWASP Top 10 2021
- **Mobile-first** — Apple HIG + WCAG 2.2 SC 2.5.5 + the project's thumb-first convention
- **Design** — the project's own `docs/design/brief.md` (Things-3 anchor, warm-earth palette, reflective Dutch tone, restrained motion, forbidden patterns)

The four agents do not see each other's findings; cross-cutting items below were synthesized after the fact.

## 1. Accessibility (WCAG 2.2 AA + Brainfog)

**Summary.** Real care for keyboard users and screen-reader semantics is visible throughout: `ScoreCircle` has a proper `role="slider"` with full keyboard alternatives (SC 2.5.7), `BottomSheet` implements focus trap + Escape + body scroll lock, the heatmap uses `role="grid"`, tabs and radiogroups are correctly labelled. Warm-earth tokens mostly hit AA contrast. The strongest surface is the login + 2FA flow (correct `autoComplete` tokens, `role="alert"` errors, justified `autoFocus`). The weakest is the `QuickEntryFlow`/`BottomSheet` pair: initial focus on Close (not on the primary action), tap targets below the 44×44 floor on three of the dialog's primary buttons, and a brainfog violation in the 500 ms score auto-advance that fires even with `prefers-reduced-motion` set. Save-status live region likely chatters at screen readers. Two contrast misses around `text-fg-subtle` and `text-accent` at small sizes.

### Findings

#### A-H1 · high · QuickEntryFlow auto-advances after 500 ms — breaks "no time pressure"
**Location:** [src/components/lab/quick-entry-flow.tsx:62, 100-111](src/components/lab/quick-entry-flow.tsx#L62)
**Evidence:**
```ts
const AUTO_ADVANCE_SCORE_TO_NOTE_MS = 500;
...
if (initialEntry === null) {
  autoAdvanceRef.current = setTimeout(() => setStep('note'), AUTO_ADVANCE_SCORE_TO_NOTE_MS);
}
```
**Impact:** Half a second after committing a score, the dialog content morphs to the note step. For a screen-reader user the content changes without warning; for a keyboard user mid-Tab focus jumps unpredictably; for a brainfog user who paused to read "7" out loud before considering a revise, the surface has already moved on. Contradicts WCAG 2.2 SC 2.2.1 (Timing Adjustable) and the brief's "no time pressure" rule. Not suppressed by `prefers-reduced-motion`.
**Recommendation:** Remove auto-advance entirely (require explicit "Volgende") — or honor `prefers-reduced-motion`, lengthen to ≥1500 ms, and add a visible "Volgende verschijnt zo…" affordance with a cancel.

#### A-H2 · high · BottomSheet initial focus lands on Close button
**Location:** [src/components/lab/quick-entry-flow.tsx:148-156](src/components/lab/quick-entry-flow.tsx#L148); [src/hooks/use-focus-trap.ts:25-27](src/hooks/use-focus-trap.ts#L25)
**Evidence:** `useFocusTrap` calls `focusables[0].focus()`. The first focusable rendered inside `QuickEntryFlow` is the Close `✕` button (the header sits above the `ScoreCircle` in DOM order).
**Impact:** Keyboard user opens the sheet and lands on Sluiten. One stray Enter and they've dismissed the dialog they just opened. WCAG 2.4.3 Focus Order + brainfog low-error-tolerance.
**Recommendation:** Either move the close button to last in DOM order (visual position preserved via `order-*`), or have `BottomSheet` accept an `initialFocusRef` that `QuickEntryFlow` points at the relevant step component depending on `startStep`.

#### A-H3 · high · Close ✕, back, "Klaar" buttons all well under 44×44
**Location:** [src/components/lab/quick-entry-flow.tsx:148-156, 187-208](src/components/lab/quick-entry-flow.tsx#L148); also settings cog [src/components/today-shell.tsx:148](src/components/today-shell.tsx#L148)
**Evidence:** `<button aria-label="Sluiten" className="rounded-md p-1 ...">✕</button>` (≈16–20 px); back/forward `px-4 py-2 text-base` (≈36–40 px).
**Impact:** WCAG 2.5.5 enhanced is 24×24; Apple HIG / brainfog floor is 44×44. On a phone with brainfog, a missed tap on "Klaar" forces re-aim — exactly the cost the brief is trying to avoid. Same problem on settings cog and tab buttons.
**Recommendation:** `min-h-11 min-w-11` (44 px) on every primary button in the popout chrome. Close ✕ becomes a dedicated 44×44 hit-target with the glyph centered. Same applied to settings cog (`p-3` around the SVG), tabs, and the "Toon meer" link.

#### A-H4 · high · Save-status live region announces ellipsis/checkmark from three concurrent reporters
**Location:** [src/components/save-status.tsx:33-58](src/components/save-status.tsx#L33); [src/components/save-status-context.tsx:36-54](src/components/save-status-context.tsx#L36)
**Evidence:** Three components (`ScoreCircle` commit, `NoteField`, `TagCategoryList`) each call `useReportSaveStatus`. Status cycles idle→saving→saved on every keystroke-settle. The live region's `aria-label` toggles between "Opslaan" and "Opgeslagen"; payload is a literal ellipsis or check character.
**Impact:** NVDA/JAWS may announce "Opslaan"/"Opgeslagen" on every cycle — every 1.5 s during a long note. WCAG 4.1.3 Status Messages says they should not interrupt; the implementation accumulates announcements. Visible payload (`…` or `✓`) is information-free without the `aria-label`.
**Recommendation:** Single off-screen live region per page, fires only on transitions worth announcing (saved-after-saving and error). Throttle to one per ~5 s. Put the readable text in a `sr-only` span; don't toggle `aria-label`.

#### A-H5 · high · BottomSheet handle is `aria-hidden` but visually interactive — semantic gap
**Location:** [src/components/lab/bottom-sheet.tsx:145-159](src/components/lab/bottom-sheet.tsx#L145)
**Evidence:** `<div ref={handleRef} aria-hidden="true" className="...cursor-grab touch-pan-y..." onPointerDown={...}>` — pointer drag dismisses, but the handle is hidden from assistive tech.
**Impact:** SC 2.5.7 Dragging Movements is technically satisfied by Escape + backdrop + X button. The real issue: a sighted keyboard/switch-control user has no signal the handle is interactive, and the visual affordance disagrees with the semantic affordance.
**Recommendation:** Either keep `aria-hidden` (accept the handle is purely cosmetic — but then widen the X button as in A-H3) — or expose the handle as `<button aria-label="Sluit-balk, tik om dialoog te sluiten">` whose tap fires `onClose`.

#### A-M1 · medium · ScoreChart points are individual tab stops; no roving-tabindex, no `aria-current="date"`
**Location:** [src/components/score-chart.tsx:165-197](src/components/score-chart.tsx#L165)
**Evidence:** Every `<g role="button" tabIndex={0}>` per day. Screen reader announces ISO date (`2026-05-29: score 7`) not the Dutch format used everywhere else.
**Impact:** Up to 90 extra tab stops on the 90-day window. No "today" indicator. Screen-reader user can't perceive the trend (the line itself is not surfaced).
**Recommendation:** Roving-tabindex (one tab stop on chart, arrows move between days). Use `formatDateDutch()` in `aria-label`. Add `aria-current="date"` to today's point. Hidden text summary of the MA trend for non-visual users.

#### A-M2 · medium · ScoreCircle has no `aria-valuetext`
**Location:** [src/components/lab/score-circle.tsx:133-141](src/components/lab/score-circle.tsx#L133)
**Evidence:** Only `aria-valuemin`/`aria-valuemax`/`aria-valuenow`. `aria-label` is bare "Score" with no range.
**Impact:** Some screen readers announce only the bare number (loses the "of 10" scale).
**Recommendation:** `aria-valuetext={'${value} van 10'}`; `aria-label="Gevoelscore van 1 tot 10"`.

#### A-M3 · medium · `text-fg-subtle` (#9c8f82) fails AA 4.5:1 on bg
**Location:** [src/app/globals.css:21](src/app/globals.css#L21); used at [src/components/today-shell.tsx:416](src/components/today-shell.tsx#L416), [src/components/timeline-view.tsx:195](src/components/timeline-view.tsx#L195), [src/app/login/2fa-setup/setup-form.tsx:155, 159](src/app/login/2fa-setup/setup-form.tsx#L155)
**Evidence:** `#9c8f82` on `#faf6f1` ≈ 3.1:1. Used on `text-xs` (12 px) labels: chart subtitle, 2FA "Sleutel" labels, "Binnenkort" badge.
**Impact:** WCAG 1.4.3 AA requires 4.5:1 for normal-size text. Token itself doesn't meet AA at the sizes it's actually used at.
**Recommendation:** Darken `--color-fg-subtle` to ~`#7b6d5f` (≈4.5:1) — or restrict to ≥18 pt / 24 px usage and audit existing call sites.

#### A-M4 · medium · `text-accent` (#b5694a) ~3.8:1 on bg — borderline at non-large sizes
**Location:** [src/app/globals.css:23](src/app/globals.css#L23); used large at [src/components/today-shell.tsx:320](src/components/today-shell.tsx#L320) (fine, 48 px); fragile if reused at body size.
**Recommendation:** Document the rule: bare `text-accent` is only for ≥24 px or bold ≥18.66 px; smaller call sites use `text-accent-active` (#834828, ~6:1).

#### A-M5 · medium · Pulse + crossfade + auto-advance partly survive `prefers-reduced-motion`
**Location:** [src/app/globals.css:79-87](src/app/globals.css#L79) (global reset); [src/components/lab/quick-entry-flow.tsx:131-135](src/components/lab/quick-entry-flow.tsx#L131)
**Evidence:** Global reset shortens `transition-duration` to 0.01 ms (good) — but the auto-advance `setTimeout` and pulse class-toggle still fire as state changes, just with no animation.
**Impact:** Reduced-motion user still sees the surface jump from score → note 500 ms after a tap. SC 2.3.3 (AAA) Animation From Interactions.
**Recommendation:** When `matchMedia('(prefers-reduced-motion: reduce)').matches`, suppress the auto-advance entirely and skip the pulse-class toggle (rely on the live-region announcement instead).

#### A-M6 · medium · Logout confirmation does not move focus to the new "Ja, uitloggen" button
**Location:** [src/components/settings-view.tsx:58-79](src/components/settings-view.tsx#L58)
**Impact:** Pressing "Uitloggen" replaces the button with a confirm block; focus falls back to `<body>`. Screen readers don't announce the prompt (it's a plain `<p>`). SC 4.1.3 + brainfog "clear undo" rule.
**Recommendation:** On state transition to `confirming`, focus the Cancel button (so destructive action requires explicit forward tab). Wrap as `role="alertdialog"` with `aria-live`.

#### A-L1 · low · Past-day card `aria-label` uses ISO date
**Location:** [src/components/today-shell.tsx:439](src/components/today-shell.tsx#L439)
**Recommendation:** Use `formatDateDutch(entry.date)` in the `aria-label` to match the rendered text.

#### A-L2 · low · Tablist has no `aria-controls`; panels have no `role="tabpanel"`
**Location:** [src/components/today-shell.tsx:167-198, 200-258](src/components/today-shell.tsx#L167)
**Recommendation:** Panels get `role="tabpanel"` + `id` + `aria-labelledby`; tabs get `aria-controls` + arrow-key navigation. WAI-ARIA APG: Tabs With Manual Activation.

#### A-L3 · low · TagCategoryList toggles use Unicode glyphs (`+/−`, `⌃`/`⌄`)
**Location:** [src/components/tag-category-list.tsx:118, 171](src/components/tag-category-list.tsx#L118)
**Recommendation:** Inline SVG chevrons sized 14–16 px to avoid glyph fallback across fonts.

#### A-I1 · info · Motion-as-communication coincides with `prefers-reduced-motion` group
**Location:** [src/components/today-shell.tsx:285-294](src/components/today-shell.tsx#L285)
**Recommendation:** Pair the pulse with a single non-stuttering live-region announcement ("Score 7 opgeslagen") — only on the post-settle saved transition.

## 2. Security (OWASP ASVS L1+L2 + Top 10 2021)

**Summary.** Overall posture is strong: cookies are `HttpOnly`/`Secure`/`SameSite=Strict`, opaque session ids via CSPRNG, parallel-refresh coalesced through an in-process lock, origin gate + per-namespace IP rate limits on every state-changing route, narrow validators on every PUT body, complete CSP/HSTS/Referrer-Policy/Permissions-Policy/X-Content-Type-Options via `next.config.ts`, no `dangerouslySetInnerHTML`, no `console.log` of secrets, Next.js at 15.5.18 (CVE-2025-29927 not exposed). The weakest area is the admin-token-on-frontend posture and the implicit single-user model — both are documented as known follow-ups, but they materially shift blast radius today.

### Findings

#### S-H1 · high · Frontend runtime holds Directus admin static token
**Location:** [src/lib/auth/stores.ts:42-55](src/lib/auth/stores.ts#L42); [src/lib/auth/directus-session-store.ts:11-19](src/lib/auth/directus-session-store.ts#L11); [docs/decisions/0005-frontend-session-persistence.md:112](docs/decisions/0005-frontend-session-persistence.md#L112)
**Evidence:**
```ts
// stores.ts:42
const adminToken = process.env.DIRECTUS_TOKEN;
...
return createDirectusSessionStore({ directusUrl, adminToken });
```
The same client is used for create / get / peek / update / delete / cleanupExpired — no scoped read-only client for lookups.
**Impact:** Any RCE / SSRF / dependency-confusion compromise of the Next.js frontend leaks an admin-equivalent Directus token. The attacker can read every `frontend_sessions` row (plaintext access + refresh tokens — pivot to full account takeover surviving password rotation), read/modify/destroy every `day_entry`, write across all collections. Single-token bypass of the entire access-control story.
**Recommendation:** Provision a dedicated Directus role + policy + user whose policy permits only CRUD on `frontend_sessions`. Mint a static token for that user, set as `DIRECTUS_TOKEN` on Fly, rotate the existing admin token. Document in a runbook. The single highest-leverage fix in the codebase.

#### S-H2 · high · `frontend_sessions` stores access + refresh tokens in plaintext
**Location:** [directus/scripts/setup-frontend-sessions.mjs:18-19](directus/scripts/setup-frontend-sessions.mjs#L18); [src/lib/auth/directus-session-store.ts:82-87](src/lib/auth/directus-session-store.ts#L82)
**Evidence:** `createItem('frontend_sessions', { id, access_token: data.accessToken, refresh_token: data.refreshToken, ... })` — no encryption envelope, no hash.
**Impact:** Combined with S-H1, a single read surfaces live access + refresh tokens that survive password rotation. Even without S-H1, a Directus DB backup / Neon snapshot share / future read-only role becomes a session-token wallet. ASVS V8.3.4.
**Recommendation:** Encrypt `access_token` / `refresh_token` with AES-256-GCM using a key in Fly secrets (separate from `DIRECTUS_TOKEN`). The only consumer is the store wrapper. Promote from ADR 0005's "when multi-user lands" to "before any second tester logs in".

#### S-H3 · high · `day_entries` has no per-user filter — multi-user expansion will cross-read
**Location:** [src/lib/api/day-entries.ts:122-132, 169-175, 259-264](src/lib/api/day-entries.ts#L122); [src/app/api/day-entries/[date]/route.ts:106](src/app/api/day-entries/[date]/route.ts#L106)
**Evidence:** Every read/write filter is keyed only by date. No grep hits for `user_created`, `user_id`, `user_updated`, or `owner` in `src/lib/`.
**Impact:** Harmless single-user today. The moment a second account logs in, both users see and overwrite each other's `day_entries`. ASVS V4.1.3 (object-level authorization) requires the app, not just the DB, to enforce ownership.
**Recommendation:** Before any second user is provisioned: add `user_created` on `day_entries`, `day_entries_tags`, `tags`, `frontend_sessions`; add explicit `filter: { user_created: { _eq: '$CURRENT_USER' } }` on every read/write call in `src/lib/api/*`; add a Vitest regression with two mocked Directus tokens asserting cross-read fails. v1.5 hard gate.

#### S-M1 · medium · Session cookie does not use the `__Host-` prefix
**Location:** [src/lib/auth/session.ts:18-25, 104-114](src/lib/auth/session.ts#L18); [src/lib/auth/pending-otp.ts:17, 115-125](src/lib/auth/pending-otp.ts#L17)
**Evidence:** Name is `gs_session` (no `__Host-`). Attributes are otherwise correct (Secure, HttpOnly, SameSite=Strict, Path=/, no Domain).
**Impact:** ASVS V3.4.3 + OWASP Cheat Sheet recommend `__Host-` for primary session cookies — the prefix enforces at the browser that the cookie must be Secure, host-only, Path=/. Not exploitable today; cheap to harden.
**Recommendation:** Rename to `__Host-gs_session` / `__Host-gs_pending_otp`. One re-login on rollout.

#### S-M2 · medium · Origin/Referer check trusts requests with neither header
**Location:** [src/lib/auth/origin-check.ts:14-26](src/lib/auth/origin-check.ts#L14)
**Evidence:** `if (!originHeader && !refererHeader) return true;` — allows curl / SSRF chain / header-stripping proxy that already has the cookie.
**Impact:** SameSite=Strict makes this hard to exploit cross-site, but it weakens defence-in-depth. ASVS V13.2.3 expects CSRF defences to fail closed.
**Recommendation:** If both headers are absent AND method is POST/PUT/PATCH/DELETE, reject. Keep lenient GET path.

#### S-M3 · medium · Pending-OTP / pending-TFA / rate-limit maps never get swept
**Location:** [src/lib/auth/pending-otp.ts:87-95](src/lib/auth/pending-otp.ts#L87); [src/lib/auth/rate-limit.ts:51-56](src/lib/auth/rate-limit.ts#L51); [src/lib/auth/stores.ts](src/lib/auth/stores.ts) — no `setInterval` anywhere
**Evidence:** Every store exposes `cleanupExpired()`/`sweep()`; grep finds zero production callers.
**Impact:** Lazy expiry on access works, but un-accessed entries persist. Resource exhaustion vector under sustained credential-stuffing burst from many IPs.
**Recommendation:** One `setInterval` at module load in `stores.ts` sweeping all four rate limiters + pending stores every 5 minutes.

#### S-M4 · medium · `getValidatedSession` lets store `peek` errors bubble — transient Directus errors become 500s
**Location:** [src/lib/auth/get-validated-session.ts:46-92](src/lib/auth/get-validated-session.ts#L46); [src/lib/auth/directus-session-store.ts:100-103](src/lib/auth/directus-session-store.ts#L100)
**Evidence:** `directus-session-store.ts peek()` re-throws non-`NOT_FOUND` errors. `get-validated-session.ts` does not wrap `store.peek(sessionId)` in try/catch. Route handlers ([src/app/api/day-entries/today/route.ts:33](src/app/api/day-entries/today/route.ts#L33), [src/app/api/day-entries/route.ts:41](src/app/api/day-entries/route.ts#L41), [src/app/api/day-entries/[date]/route.ts:48](src/app/api/day-entries/[date]/route.ts#L48)) have no try/catch.
**Impact:** Directus 502 / network blip during `peek` returns a 500. Contradicts the eviction policy comment block ("preserve the row so the next request retries"). ASVS V7.4.1.
**Recommendation:** Wrap the `peek` call in try/catch; on network/Directus error return null (callers already treat as unauthenticated).

#### S-M5 · medium · `GET /api/day-entries` (range read) has no rate limit
**Location:** [src/app/api/day-entries/route.ts:26-71](src/app/api/day-entries/route.ts#L26); contrast PUT at [src/app/api/day-entries/[date]/route.ts:35-42](src/app/api/day-entries/[date]/route.ts#L35)
**Evidence:** Range read has origin check + session check + 90-day cap but no `dayEntryWriteRateLimiter.check(ip)`. `GET /api/day-entries/today` also has none.
**Impact:** Valid session can hammer GET on ~89-day windows freely. Single-user threat small; symmetry with PUT (which IS rate-limited) is the defense-in-depth ask. ASVS V11.1.2.
**Recommendation:** Permissive read limiter (e.g. 120/min) shared between `/api/day-entries` and `/api/day-entries/today`.

#### S-L1 · low · CSP allows `'unsafe-inline'` in `script-src`
**Location:** [next.config.ts:12-30](next.config.ts#L12)
**Recommendation:** Evaluate nonce-based CSP for inline `<script>` when upgrading Next 15.x; drop `'unsafe-inline'` from `script-src` once in place. `style-src 'unsafe-inline'` is unavoidable for Tailwind v4 — accept and document.

#### S-L2 · low · Login rate limit is per-IP only
**Location:** [src/lib/auth/stores.ts:26-29](src/lib/auth/stores.ts#L26)
**Recommendation:** When multi-user lands, add a per-email rate limiter (5/5min) in parallel with per-IP.

#### S-L3 · low · Pending-OTP cookie not cleared on successful non-2FA login
**Location:** [src/app/api/auth/login/route.ts:62-84](src/app/api/auth/login/route.ts#L62)
**Recommendation:** Append `buildPendingOtpCookie(null, 0)` to the success Set-Cookie list. One line.

#### S-L4 · low · OTP attempt counter is per-`pendingId`, not per-IP — chain-bypassable
**Location:** [src/app/api/auth/login/verify/route.ts:62-72](src/app/api/auth/login/verify/route.ts#L62)
**Evidence:** Each `/api/auth/login` creates a fresh pending entry with `attempts: 0`; rate limit is 5/5min/IP.
**Impact:** Effectively 3 × 5 = 15 OTP attempts per 5-min IP window. ~0.0015% per window — below brute-force feasibility, but the constant name lies.
**Recommendation:** Either rename `OTP_ATTEMPT_CAP` to `_PER_PENDING`, or add a per-IP OTP attempt counter (limit 6/5min) that survives across pending entries.

#### S-I1 · info · Public `/over` route exempted from middleware — verified safe
**Location:** [src/middleware.ts:36](src/middleware.ts#L36); [src/app/over/page.tsx](src/app/over/page.tsx)
**Recommendation:** No change. If `/over` ever takes form input, reinstate origin + rate-limit gates.

#### S-I2 · info · `X-Frame-Options` not present alongside CSP `frame-ancestors`
**Location:** [next.config.ts:17-35](next.config.ts#L17)
**Recommendation:** Optional belt-and-braces for very old browsers: add `X-Frame-Options: DENY`.

### Positives

- Cookie attributes textbook: HttpOnly + Secure + SameSite=Strict + Path=/ + Max-Age. Opaque-id-only — JS never sees the access token.
- Session id via `crypto.randomUUID()` (CSPRNG, 122-bit) in every store path.
- Parallel-refresh race coalesced through an in-process `Map<sessionId, Promise>` with self-cleanup. Eviction policy distinguishes terminal (`invalid_refresh_token`) from transient (`network_error`/`directus_error`) — exactly right.
- Every state-changing route runs origin check + IP rate limit + session validation in a consistent order. No drift between handlers.
- Input validators are explicit, narrow, tested: `validateScore` 1–10 integer; `normalizeNote` 10 000 char cap; `validateTagIds` array of strings + no dupes; `validateDate` YYYY-MM-DD calendar + not-future.
- Pending-OTP store hardened: overwrite-with-random-bytes before eviction; non-enumerable `toString` to defeat accidental `console.log`.
- TFA secret binding server-side: `/2fa/enable` looks up the secret stashed by `/2fa/generate` keyed by `sessionId`, ignoring any client-sent `secret`. Audit H2 correctly applied.
- Security headers complete in one place — CSP with `frame-ancestors 'none'` + `base-uri 'self'` + `form-action 'self'`, HSTS, X-Content-Type-Options, Referrer-Policy, Permissions-Policy.
- Next.js 15.5.18 — patched against CVE-2025-29927.
- No `dangerouslySetInnerHTML` anywhere. No `target="_blank"` without `rel`.
- No secrets hardcoded; every Directus URL/token goes via `process.env`; internal Fly network used for server-side Directus.
- Error boundary deliberately renders generic message — no PII leak.
- Logout idempotent; hits Directus to invalidate the refresh token before clearing cookie.

## 3. Mobile-first (iOS PWA + Apple HIG + thumb-first)

**Summary.** Strongest area is the `BottomSheet` primitive itself — focus trap, body scroll lock, visualViewport keyboard dodge, safe-area-inset-bottom padding, pointer-capture — all done correctly. Weakest area is iOS PWA installability (no manifest, no apple-touch-icon, no `apple-mobile-web-app-capable`, no `viewport-fit=cover`-aware layout padding outside the sheet). Systemic risks: the 100vh trap on every shell, several sub-44 px touch targets in invoer zones (tag chips, settings cog, sheet close), the tab bar at the top (against iOS convention and the project's own thumb-first rule), and unscoped `hover:` styles that latch as sticky highlights on iOS.

### Findings
